# Architecture Exploration — 001_playing_agent

**Task:** "Find the bakery and list what is on the menu."
**Models used in this session:** Haiku 4.5 (initial attempts) → Sonnet 5 (default, completed the task) → Fable 5 (session ended before further work; see separate Fable 5 exploration doc for an independent run of the same prompt).
**Result:** Task completed successfully. Bakery located in Midgaard (Main Street, west of Market Square, then north); full menu (danish pastry 7, bread 14, waybread 72) retrieved and recorded in `data/player.md` / `data/world.md`.

## Timeline

1. **Haiku 4.5** — Attempted the MUD login by batching all commands (username, password, `look`) into a single `nc` invocation with `sleep` delays between echoed lines, with no inspection of intermediate server output. Two consecutive attempts along this line (a multi-line heredoc `nc` session, then a Python `socket` script written to the scratchpad) were rejected by the user before either could run.
2. User redirected via `AskUserQuestion` to a slower, observable approach: send one command at a time via `nc` and inspect the response before sending the next.
3. **Sonnet 5** (switched by the user before answering the question) — Rebuilt the connection using a named pipe (`mkfifo`) plus a backgrounded `nc` process, with a long-lived keep-alive writer process to hold the pipe open across separate tool calls (each `Bash` invocation is a fresh shell; a naive `echo > fifo` per call closes its writer and delivers EOF to `nc`, killing it — this cost one dead connection and a server-side login timeout before the keep-alive trick was added). After that, the login flow (name → password → press-return → menu choice) was driven one step at a time, reading output after each send, and completed cleanly.
4. Explored the map (Common Square → Market Square → Main Street east, dead end → backtrack → Main Street west → Bakery), used `list` in the bakery to retrieve the menu, and wrote the results to `data/player.md` and `data/world.md` as instructed by `CLAUDE.md`.

## Observations

- **Temporary connection scripts were created.** A Python socket script (`mud_connect.py`) was written to the scratchpad but rejected before execution. The eventual working solution did not persist a script file but did create ephemeral shell-level state (a named pipe, a backgrounded `nc` process, and a backgrounded `while true; do sleep 3600; done` keep-alive writer) to work around the fact that shell state/file descriptors do not persist between separate `Bash` tool calls.
- **These temporary artifacts were never cleaned up.** `/tmp/mud_in` (fifo), `/tmp/mud_out`, `/tmp/mud_pid`, `/tmp/mud_keepalive_pid`, and the backgrounded `nc`/keep-alive processes were all still running at the time this document was written, well after the task had completed. The agent did not proactively `kill` the processes or remove the files once the goal was reached.
- **The agent struggled with the deterministic MUD login flow.** The login sequence (name → confirm/password → press-return → menu) is stateful and timing-sensitive (the server issued a hard login timeout — "Timed out... goodbye" — on one attempt where the connection sat idle too long before the username was sent). Blind batching of commands without reading intermediate output was the direct cause of the early failures; this was a process/approach issue rather than a language-model reasoning failure per se.
- **Unrelated repository files were not read during the original task.** Contrary to the general expectation that agents drift into unnecessary exploration, this run only read `data/player.md` and `data/world.md` — both explicitly named in `CLAUDE.md` as the state files to maintain. No other repository files (README, CHALLENGES.md, source code, etc.) were read while pursuing the bakery goal. (Broader repository exploration did happen afterward, but only as part of a *separate*, later request to figure out where to place this very document — not as part of the original task.)
- **The agent had poor visibility into its failed login attempts, early on.** The first two Haiku-4.5-era attempts sent multiple commands without reading server responses in between, so failures (e.g., a garbled name registration where the server appeared to echo back "Look" instead of "dummy" as the candidate name) were not diagnosed — they were simply retried with minor variations until the user intervened. Visibility improved substantially once the one-command-at-a-time pattern was adopted; from that point on, every send was followed by a read before the next action was chosen.
- **Whether increasing model intelligence improved the result is confounded in this session.** The switch from Haiku 4.5 to Sonnet 5 happened at the same moment the user also supplied explicit process guidance (via `AskUserQuestion`: "retry with nc, one command at a time"). The subsequent success cannot be cleanly attributed to the model change alone — the two rejected Haiku 4.5 attempts were stopped by the user before they could finish executing, so it's unknown whether they would have failed on their own merits or simply needed the same slow-down. The root cause of the early struggle (batched, unobserved sends against a stateful, timeout-sensitive login prompt) is a control-flow/tool-usage issue that a more capable model is not guaranteed to avoid without also adopting a read-before-write discipline. A same-prompt run on a different model in an independent session (tracked separately) is a better basis for isolating the effect of model capability from the effect of user-supplied process guidance.
